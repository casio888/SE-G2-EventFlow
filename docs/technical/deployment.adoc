=== System lokal starten

[source,bash]
----
# 1. ssh key erstellen für commits mit korrektem Account
ssh-keygen -C "<name>" -f .ssh/seproject_ed25519 -t ed25519

# 2. ssh config für git anlegen
tee -a .ssh/config <<'EOF'

Host github.com
  Hostname github.com
  User git
  IdentityFile ~/.ssh/seproject_ed25519
  IdentitiesOnly yes
EOF

# In github.com -> Einstellungen -> SSH und GPG Keys -> New SSH Key
# Dort den Inhalt vom Public Key hinzufügen
cat ~/.ssh/seproject_ed25519.pub

# Testen (Erfolg steh in der Nachricht. Session wird immer direkt beendet)
ssh github.com



# 3. Repository klonen
git clone git@github.com:casio888/SE-G2-EventFlow.git
cd SE-G2-EventFlow/deployment

# 4. Links anlegen
ln -s -f docker-compose.yml.dev docker-compose.yml
cp docker-compose.yml.dev.example docker-compose.yml.dev

# 5. Anpassen der Pfad-Platzhalter <>, in dockerfile
pwd
vim docker-compose.yml.dev

# 6. Umgebungsvariablen konfigurieren
cp .env.django.example .env.django
# 7. Anpassen von Variablen in Platzhaltern <>. Secret key kann beliebig gewählt werden. SQL Password muss mit postgres_db_password.txt übereinstimmen
vim .env.django
echo "<password>" > postgres_db_password.txt

# 8. Docker container bauen (nur 1 mal zu Beginn oder bei Änderungen an Dockerfiles)
docker compose build

# 9. Docker Stack starten (-d startet container detached von der shell. Weniger für debugging geeignet.)
docker compose up

# 10. Prüfen, ob alles läuft
# Frontend:  http://localhost:8000
----

=== System stoppen und aufräumen

[source,bash]
----
# Dienste stoppen (Daten bleiben erhalten)
cd deployment

----

=== Häufige Probleme beim lokalen Start

[cols="2,3", options="header"]
|===
| Problem | Lösung

| Port bereits belegt (`address already in use`)
| `docker compose down` ausführen oder belegten Port prüfen:
  `lsof -i :8080` (macOS/Linux) bzw. `netstat -ano` (Windows)

| Datenbank-Container startet nicht
| Volumes löschen, inhalt leeren und neu starten: `docker compose down -v && suro rm -rf deployment/* && docker compose up`

| Django kann keine Verbindung zu PostgreSQL aufbauen
| Prüfen ob `.env.django`-Datei vorhanden ist; 
|===

== CI/CD-Pipeline

Die Pipeline automatisiert Qualitätsprüfung und Deployment. Sie stellt sicher,
dass kein Code in `main` gelangt, der Tests bricht oder Qualitätsstandards verletzt.

Noch nicht implementiert
////
=== Pipeline-Schritte im Detail

[cols="1,2,1,1", options="header"]
|===
| Schritt | Beschreibung | Schlägt fehl bei | Ausgeführt bei

| Linting
| Code-Stil und statische Analyse prüfen
| Stilfehler oder Warnungen über Schwellwert
| Jedem Push

| Unit-Tests
| Alle Unit-Tests ausführen
| Fehlschlagende Tests
| Jedem Push

| Integrationstests
| Tests gegen echte Datenbank (Testcontainers)
| Fehlschlagende Tests
| Jedem Push

| Abdeckungsprüfung
| Testabdeckung messen und mit Mindestwert vergleichen
| Unterschreitung des Mindestwerts (siehe Testkonzept)
| Jedem Push

| Image-Build
| Docker-Images für alle Services bauen
| Build-Fehler
| Jedem Push auf `main` und PRs

| Staging-Deployment
| Neue Images auf Staging-Umgebung ausrollen
| Deployment-Fehler oder fehlschlagende Smoke Tests
| Merge in `main`

| Produktions-Deployment
| Neues Release auf Produktion ausrollen
| Deployment-Fehler oder fehlschlagende Smoke Tests
| Manuell nach Tag-Erstellung
|===

Die vollständige Pipeline-Konfiguration befindet sich in:

----
.github/workflows/
├── ci.yml          ← Qualitätsprüfung (bei jedem Push)
└── deploy.yml      ← Deployment (Staging & Produktion)
----


== Deployment-Ablauf

=== Deployment auf Staging (automatisch)

Jeder Merge in `main` löst automatisch ein Deployment auf Staging aus.
Kein manueller Eingriff erforderlich.

. Pipeline baut neue Docker Images
. Images werden in die Container-Registry gepusht
. Staging-Umgebung zieht die neuen Images
. Smoke Tests werden automatisch ausgeführt
. Team wird bei Erfolg oder Fehler benachrichtigt
////

=== Release und Deployment auf Produktion (manuell)

[source,bash]
----
# 1. In Github einen Tag für den Main Branch erstellen. Tags nur auf Main!
git checkout 1.0.0

# 2. Deployment auf Produktion
cd deployment
docker compose up -d --build

----

NOTE: Wir verwenden _Semantic Versioning_ mit dem Schema MAJOR.MINOR.PATCH: 
MAJOR bei inkompatiblen Änderungen,
MINOR bei neuen Features (abwärtskompatibel),
PATCH bei Bugfixes.

=== Smoke Tests nach Deployment

Nach jedem Deployment werden folgende Prüfungen ausgeführt:

[cols="2,2", options="header"]
|===
| Prüfung | Erwartetes Ergebnis

| Startseite laden (Frontend)
| HTTP 200, Ladezeit < 1 s (F12->Network)

| Login-Flow
| Erfolgreicher Login mit Testnutzer möglich
|===

== Rollback

Ein Rollback ist notwendig, wenn ein Deployment kritische Fehler in der
Produktionsumgebung verursacht. Schnelles Handeln minimiert die Auswirkungen.

=== Wann wird ein Rollback durchgeführt?

* Kritische Fehler in Produktion nach einem Deployment
* Smoke Tests schlagen nach Deployment fehl
* Fehlerrate oder Antwortzeiten steigen signifikant an

=== Rollback-Ablauf

. *Entscheidung treffen* — Tech Lead oder Product Owner entscheiden über Rollback
. *Stakeholder informieren* — Team und relevante Stakeholder benachrichtigen
. *Rollback ausführen* (eine von zwei Optionen):

[source,bash]
----
# Option A: Letztes stabiles Tag erneut deployen
# → GitHub → Actions → deploy.yml → Version des letzten stabilen Tags eingeben

# Option B: Manuell das vorherige Image aktivieren
cd deployment
docker compose pull <vorherige Version>
docker compose down
docker compose up -d
----

[start=4]
. *Smoke Tests ausführen* — sicherstellen dass das System wieder stabil ist
. *Ursache analysieren* — was hat das Problem verursacht?
. *Nachbereitung dokumentieren* — im Ticket oder als Kommentar im ADR
