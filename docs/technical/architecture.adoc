== Architekturentscheidungen

=== ADR-001: Python als Backend-Programmiersprache

[cols="1,3"]
|===
| *Status*
| Akzeptiert

| *Datum*
| 19.5.2026

| *Kontext*
| Um ein Framework auszuwählen muss eine Programmiersprache für das Backend festgelegt werden.

| *Entscheidung*
| Wir verwenden Python, weil dafür bei allen Teammitgliedern Grundkenntnisse vorhanden sind. Zusätzlich ist der Tech-Lead sehr erfahren mit Python-Programmierung (und Webprogrammierung) und kann dadurch besser aushelfen (ausführlichere Reviews, gezielte Spikes zur Verständnissbildung)

| *Alternativen*
| Java: Auch bei Java gab es bei allen Teammitgliedern Grundlagen. Dafür spricht die Typisierung der Sprache und gute bessere Möglichkeiten Parallelisierung umzusetzen. Dagegen sprechen das etwas kompliziertere Setup auf Serverseite, keine zwingende Notwendigkeit für "high performance" und eine leichte Unbeliebtheit von Java beim Team.

JavaScript/Typescript: Express.js oder Nuxt.js wären Optionen für Javascript Backends. Dafür sprechen umfangreiche Ökosysteme mit Vorlagen und viel Scaffolding. Dagegen spricht, dass nicht alle Teammitglieder mit JS/TS vertraut sind.

| *Konsequenzen*
| Lock in auf Python basierte Backends.
|===

=== ADR-002: Backend Framework

[cols="1,3"]
|===
| *Status*
| Akzeptiert

| *Datum*
| 19.05.2026

| *Kontext*
| Der Umfang des Projektes erfordert, dass ein Framework für die Entwicklung des Backends verwendet wird. Damit soll Struktur vorgegeben werden.

| *Entscheidung*
| Wir verwenden Django, weil dieses vom Umfang der eingebauten Features (ORM, Auth, App-Struktur zur trennung von Anwendungsbreichen) viel von dem Mitbringt, was wir benötigen. Des weiteren gibt es ein umfassendes Ökosystem mit Beispielen und Prinzipien zur Umsetzung von Probleme, die wir erwarten. Zusätzlich hat der Tech-Lead Erfahrung mit dem Framework sammeln können und kann damit (hoffentlich) einne schnelleren Einstieg in das Framwork garantieren.

| *Alternativen*
| FastAPI: Ähnliche Features wie Django, allerdings fehlen voreingebaute Authentication und Logging Lösungen. Außerdem ist die Community etwas kleiner und es bestehen keine Vorerfahrungen.

Flask: Ist sehr lightwheight, aber gibt damit auch wenig Struktur vor, die für ein Projekt wie unseres sehr hilfreich ist. Keine eigebautes ORM bedeutet die Anbindung oder Eigenentwicklung einer Datenbankanbindung.

| *Konsequenzen*
| Projektstruktur ist an Django-App Struktur gebunden und damit sehr schwer auf andere Frameworks umzubauen.
|===

=== ADR-003: Datenbanksystem

[cols="1,3"]
|===
| *Status*
| Akzeptiert

| *Datum*
| 01.06.2026

| *Kontext*
| Die persistente Speicherung von Daten ist über eine Datenbank abzudecken. Dateibasierte Speicherung bietet sich nicht an, beim Projektumfang. Mit Ausblick auf die User Stories empfiehlt sich eine relationale Datenbank.

| *Entscheidung*
| PostgreSQL, da es ein weitverbreitetes Open-Source System ist. Es implemetiert die ACID-Regeln sehr streng. sehr Zudem ist es eine relationale Datenbank. Das wohl wichtigste Feature von Postres ist, dass es bereits eine Einbindung in Django gibt. 

| *Alternativen*
| MariaDB: Es bietet die Standardkonformen Befehle von SQL und lässt sich ebenfalls sehr gut in Programme einbinden. Ein weiterer Vorteil ist, dass es Open-Source ist. Es bietet aber weniger Features als Postgres zudem gibt es eine größere Communitie von Postgres in kombination mit Django.

  MSSQL: Es funktioniert besonders gut in Kombination mit anderen Microsoftprodukten. Dabei Unterstützt es ebenfalls den Standard von SQL. Da es auf Microsoft optimiert ist, unser System aber auf Linux läuft haben wir uns dagegen entschieden.
  
  OracleDB: Ist in der Industrie weit verbreitet und hat einen guten Support, allerdings ist es kostenpflichtig, weshalb es für uns nicht in Frage kommt.

| *Konsequenzen*
|Wir sind an diese Relationale Datenbank gebunden, da ein späterer Wechsel auf eine andere Datenbank einen großen Aufwand bedeutet. 
Das Team muss sich in die Datenbank hineinarbeiten, dafür ist es weitverbreitet und Open-Source, wodurch man viele Informationen findet. 
|===

=== ADR-004: Frontend Framework

[cols="1,3"]

|===

| *Status*
| Akzeptiert

| *Datum*
| 19.05.2026

| *Kontext*
| Um Barrierefreiheit zu unterstützen und die Entwicklung des Frontends durch Building Blocks zu beschleunigen, bietet es sich an, ein Frontend Framework zu verwenden.

| *Entscheidung*
| Vorerst wird im Frontend Bootstrap genutzt. 
Dadurch soll eine einfache und agile Umsetzung von Layout und UI gewährleistet werden. +
Da Bootstrap schlicht ist, bleibt der Aufwand für das Frontend gering. +
Auch ist die Hürde des Erlernens für das Team niedrig. 

| *Alternativen*
| Vue in Kombination mit Vite wäre eine komponentenbasierte Lösung. 
Da diese aber einen Overhead mit sich bringt und kein Teammitglied Erfahrung mit Vue hat, ist das Risiko vorerst zu groß. 
Wenn das Frontend deutlich komplexer wird, wäre dies aber eine Grundlage zur Neuevaluierung.
React wäre ebenfalls eine Lösung, da dieses Framework aber einen erheblichen Lernaufwand mit sich bringt, wurde dies verworfen.

| *Konsequenzen*
| Das Frontend bleibt vorerst simpel und schnell umsetzbar. 
Es besteht kein zusätzlicher Build-Prozess durch Vite o. React.
Bootstrap vereinfacht das Entwickeln des Layouts. 
Komplexere Logik muss in Vanilla JS umgesetzt werden. 
Falls das Frontend erheblich interaktiver bzw. komponentenlastig wird, ist eine erneute Diskussion sinnvoll. 
Die Entscheidung minimiert kurzfristig das Risiko für Zeitfehlmanagement durch übermäßige Einarbeitungszeit.

|===

=== ADR-005: Produktions Deployment Prozess

[cols="1,3"]
|===
| *Status*
| Akzeptiert

| *Datum*
| 31.05.2026

| *Kontext*
| Um Probleme im Deployment der Anwendung zu reduzieren ist ein einheitlicher Prozess notwendig.

| *Entscheidung*
| Wir nutzen Docker + Docker Compose für den Deployment Prozess. Die Dockerfiles geben einheitliche Buildanleitungen vor, die auch bei der Lokalen Entwicklung wiederverwendet werden können.
Außerdem besteht beim Tech Lead viel Vorerfahrung mit der Nutzung von Docker.

| *Alternativen*
| Manuelles Deployment auf dem Server ("baremetal") hat den Nachteil, dass Config Files schlechter im Git getrackt werden können und später meistens unklar ist welche Files vom Projekt modifiziert wurden und welche mit Installationen als Default mitkommen. Außerdem gibt es mehr Schritte im Prozess, was zu mehr Fehlern führen kann.

Ansible wäre eventuell eine Alternative zu Docker, allerdings wäre dafür mehr Overhead notwendig und es besteht keine Vorerfahrung im Team.

| *Konsequenzen*
| Schreiben und Pflegen von Dockerfiles. Buildprozess der Docker images muss bei redeploys jedes Mal ausgeführt werden.
|===

=== ADR-006: Webserver

[cols="1,3"]
|===
| *Status*
| Akzeptiert

| *Datum*
| 31.05.2026

| *Kontext*
| Django braucht einen Webserver, da der interne Developmentserver nicht sicher ist.

| *Entscheidung*
| Wir nutzen Nginx als Reverse Proxy und gunicorn als WSGI server.
Es ist eine bewährte Kombination, es gibt viel Community Support. Aber es ist im Team keine Vorerfahrung vorhanden.

| *Alternativen*
| Apache (mit mod_wsgi) ist ebenfalls Valide als Alternative zu Nginx. Apache ist etwas imperformanter bei höheren Lasten, die wir aber nicht erwarten. Hier gab es keinen klaren Favoriten und die Entscheidung wurde vom Teach Lead getroffen.

uWSGI ist eine Alternative für gunicorn. uWSGI ist in der Konfiguration aber etwas komplexer.

Uvicorn nutzt ASGI anstatt wsgi. Da es keine für uns relevanten Vorteile bringt und die Kombination mit gunicorn mehr Community Supportt besitzt, wurde sich dagegen entschieden.

| *Konsequenzen*
| Trennung von Reverse Proxy und Anwendungsserver (wsgi). Keine Unterstürtzung von Async und WebSockets (wsgi anstatt asgi)
|===
